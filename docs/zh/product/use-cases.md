# Plato 产品用例

Plato 适合这样的场景：用户知道自己想要的结果，但希望系统在行动前先把工作结构展示出来。

核心循环是：

```text
目标 -> 计划 -> 检查 -> 执行 -> 查看结果和审计
```

## 1. 规划个人网站

用户目标：

> 帮我做一个简单的个人网站。

Plato 应该做什么：

- 在信息不足时追问；
- 生成内容、布局、实现、检查等任务；
- 允许用户在发布前修改任务；
- 执行后展示结果和文件变化。

为什么适合 Plato：

用户可能不知道页面、技术栈和执行顺序。可见的任务计划比一段长回答更容易检查和调整。

## 2. 把粗略想法拆成任务

用户目标：

> 我想做一个小工具，收集并总结网上资料。

Plato 应该做什么：

- 明确资料来源、输出格式和执行频率；
- 拆成规划、实现、验证、总结等任务；
- 在执行前暴露假设。

为什么适合 Plato：

难点不是给一个答案，而是让用户判断系统准备做的工作是否正确。

## 3. 生成课件或学习材料

用户目标：

> 做一个七年级生物 HTML 课件。

Plato 应该做什么：

- 询问受众、章节和教学风格；
- 生成大纲、页面、例子、测验、检查等任务；
- 把结果和任务、证据关联起来。

为什么适合 Plato：

非技术用户可以从教学目标出发，让 Plato 把目标转成结构化工作。

## 4. 准备安全的文件修改计划

用户目标：

> 我想整理这个项目，但不希望误改文件。

Plato 应该做什么：

- 在可用时检查上下文；
- 修改前先提出任务；
- 对高风险动作暂停确认；
- 让用户事后检查文件变化和审计证据。

为什么适合 Plato：

这种场景最需要 Control Plane 和 Trust Plane：用户既要知道将要发生什么，也要知道事后发生了什么。

## 5. 检查结果是否可以接受

用户目标：

> 看看改动是否足够好，是否还需要继续。

Plato 应该做什么：

- 总结任务结果；
- 展示文件变化、警告和缺失证据；
- 建议接受、修改、重试或停止。

为什么适合 Plato：

产品不应该只停在 "完成"。它应该帮助用户判断结果是否可信、是否值得继续。

## 6. 将 bug 修复拆成可检查步骤

用户目标：

> 修复这个 bug，但不要破坏无关行为。

Plato 应该做什么：

- 拆出调查、修复、测试、复核任务；
- 缺少上下文时追问；
- 修复后保留审计线索。

为什么适合 Plato：

bug 修复需要边界和证据。用户要理解改了什么，也要理解为什么这样改。
